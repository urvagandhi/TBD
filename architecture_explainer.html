🧠 What is CrewAI?

CrewAI is a framework that lets you create multiple AI agents and make them work in sequence. Each agent gets the output of the previous one automatically. Think of it like an assembly line — each worker does one job and passes the part to the next.

🔮 What LLM do we use?

Gemini 2.0 Flash via Google AI API. We set temperature=0 on every call — this means the LLM gives consistent, deterministic output every time. No random variation.

⚡ No Model Training

We do NOT train any model. We do NOT fine-tune anything. We just use Gemini 2.0 Flash as a reasoning engine — it already knows how academic papers are structured from its training data.

📋 Rules as JSON Files

All journal formatting rules live in simple JSON files. APA has its own file, IEEE has its own, etc. Agents READ these files and apply the rules. To add a new journal, just add a new JSON file.

The 5-Agent Pipeline

Before Agents Run
🌐 FastAPI receives your upload
User uploads PDF or DOCX via React frontend → FastAPI validates file type and size (max 10MB) → If PDF: PyMuPDF extracts all text → If DOCX: python-docx extracts all text → Text string + journal name passed to crew.kickoff()
01
Ingest Agent Label & Structure Raw Text click to expand ↓
Raw text from PDF is just a wall of text — no structure, no labels. This agent reads it and adds structural markers so every other agent can understand the paper.
What comes IN?
Plain text string. Example: paper_content = "Deep Learning for NLP: A Survey\nJohn Smith, Jane Doe\nAbstract\nIn this paper..." — Just raw extracted text. No formatting info.
Does it call the LLM?
YES. The LLM reads the raw text and adds labels. Why? Because figuring out what's a heading vs body text vs a reference is a language understanding task. Rule-based regex alone would fail on edge cases.
What does it ask the LLM?
"Here's a research paper as raw text. Add these labels inline: [TITLE], [AUTHORS], [ABSTRACT_START], [ABSTRACT_END], [HEADING_H1:text], [HEADING_H2:text], [CITATION:text], [REFERENCE_START] etc. Do NOT change any original text."
What comes OUT?
Same text but with labels added. Example below. This output is automatically passed to Agent 2 by CrewAI.
# Output example (labelled text string): [TITLE] Deep Learning for NLP: A Survey [AUTHORS] John Smith, Jane Doe, Robert Johnson [ABSTRACT_START] In this paper, we present a comprehensive survey of deep learning techniques for natural language processing... [ABSTRACT_END] [KEYWORDS] Keywords: deep learning, NLP, transformers, BERT [HEADING_H1:Introduction] Recent advances in AI [CITATION:(LeCun et al., 2015)] have led to significant improvements in language understanding... [HEADING_H2:Background] ... [REFERENCE_START] [1] LeCun, Y., et al. (2015). Deep learning. Nature...
02
Parse Agent Extract Structured Information click to expand ↓
Takes the labelled text and extracts every piece of information as structured JSON — title, authors, abstract word count, all sections, citations, references, and more.
What comes IN?
The labelled text string from Agent 1. CrewAI passes it automatically as context.
Does it call the LLM?
YES. Because extracting structured data from text (counting words, identifying citation styles, detecting IMRAD structure, matching citations to references) is a language understanding task that needs reasoning, not just regex.
What does it ask the LLM?
"Parse this labelled paper and return a complete JSON structure with title, authors, abstract (with word count), IMRAD booleans, all sections with headings + levels + citations found in them, all figures, all tables, complete reference list."
What comes OUT?
A strict JSON object — the paper_structure. This is the most important intermediate output. Everything else depends on this being accurate.
paper_structure.json
{ "title": "Deep Learning for NLP: A Survey", "authors": ["John Smith", "Jane Doe"], "abstract": { "text": "In this paper, we present...", "word_count": 267 ← APA limit is 250 → violation! }, "imrad": { "introduction": true, "methods": true, "results": true, "discussion": false ← missing! }, "sections": [ { "heading": "Introduction", "level": 1, "in_text_citations": ["(LeCun et al., 2015)", "[3]"] } ], "references": ["LeCun, Y. et al. (2015)..."], "metadata": { "citation_style_detected": "mixed", ← problem! "total_references": 42 } }
03
Interpret Agent Load Journal Formatting Rules click to expand ↓
Loads the complete formatting rulebook for the selected journal from a JSON file. No LLM needed — just a fast file lookup.
What comes IN?
The journal_style string from the original user input — e.g. "APA 7th Edition" or "IEEE". Note: this comes from the original crew.kickoff() inputs, NOT from Agent 2.
Does it call the LLM?
NO — for supported journals. It simply calls rule_loader.load_rules("APA 7th Edition") which reads rules/apa7.json from disk. The LLM is only used to format the output for the pipeline.
Why not just read the JSON directly?
Because it's an agent in a CrewAI pipeline — it needs to produce output in a format the next agent can consume as context. The agent wraps the JSON loading in a @tool function.
What comes OUT?
The exact content of the rules JSON file — all 11 sections: document settings, abstract rules, heading styles, citation format, reference format, figure/table rules, etc.
rules/apa7.json (output of Interpret Agent)
{ "style_name": "APA 7th Edition", "document": { "font": "Times New Roman", "font_size": 12, "line_spacing": 2.0 ← double spaced }, "abstract": { "max_words": 250, "label_bold": true, "label_centered": true }, "headings": { "H1": { "bold": true, "centered": true, "case": "Title Case" } }, "citations": { "style": "author-date", "format_three_plus": "(Smith et al., 2020)" }, "references": { "ordering": "alphabetical", "hanging_indent": true } // ... 6 more sections }
04
Transform Agent Find Violations & Fix Everything click to expand ↓
This is the core agent. It compares what the paper has against what the journal requires, builds a complete list of fixes, and produces instructions for creating the formatted DOCX.
What comes IN?
paper_structure from Agent 2 (what the paper has) + rules JSON from Agent 3 (what the journal requires). Both arrive automatically via CrewAI context.
Does it call the LLM?
YES — this is the most complex LLM call. Comparing two complex structures, understanding semantic equivalence ("Conclusion" = "Discussion"), reformatting citations, reordering references — this requires language intelligence.
What exactly does it compare?
Font name, font size, line spacing, abstract label format, abstract word count vs limit, heading bold/italic/centered/case for each level, citation format (author-date vs numbered), two-author format, three+ author (et al.), reference ordering, figure/table caption positions and label formats.
What is docx_instructions.sections?
This is a complete ordered array of every section in the document — title, abstract, every heading, every body paragraph, every figure/table caption, every reference entry — each tagged with the correct formatting to apply. This is what python-docx uses to build the output file.
transform_result.json (output of Transform Agent)
{ "violations": [ { "element": "font", "current_state": "Arial 11pt", "required_state": "Times New Roman 12pt", "severity": "high" }, { "element": "abstract word count", "current_state": "267 words", "required_state": "max 250 words", "severity": "medium" } ], "changes_made": [ "Font changed: Arial → Times New Roman", "14 citations reformatted to APA author-date", "References reordered alphabetically" ], "citation_replacements": [ { "original": "[3]", "replacement": "(Wang et al., 2019)" } ], "docx_instructions": { "font": "Times New Roman", "font_size": 12, "sections": [ ← REQUIRED: full doc in reading order { "type": "title", "text": "Deep Learning...", "bold": true }, { "type": "heading", "level": 1, "text": "Introduction", "bold": true, "centered": true, "case": "Title Case" }, { "type": "body", "text": "Recent advances..." }, { "type": "reference_entry", "text": "LeCun, Y...", "hanging_indent": true } ] } }
05
Validate Agent Run 7 Checks, Score Compliance click to expand ↓
Independently verifies the formatted document meets all requirements. Runs 7 specific checks and produces a per-section score from 0–100.
What comes IN?
paper_structure + transform_result + rules — all passed automatically via CrewAI context chain.
Does it call the LLM?
YES. Cross-referencing citations to references, detecting reference age from years, identifying self-citations by author name matching — these are language tasks requiring reasoning.
Why validate if Transform already fixed everything?
Transform fixes formatting. Validate checks correctness of the content relationships — did every [1] get matched to a reference? Are figures numbered sequentially? Did the abstract word count get reported accurately? Two different jobs.
What MUST always be in the output?
overall_score is mandatory. crew.py raises a hard ValueError if it's missing. No fallback. This is intentional — if we can't score the document, we should not pretend we succeeded.
After Agents Finish
📁 crew.py writes the DOCX file
crew.py calls _write_docx_from_transform(transform_result) → docx_writer reads docx_instructions.sections array → python-docx creates every paragraph with correct font/bold/spacing → saves to outputs/formatted_xxxx.docx → FastAPI returns download_url + compliance_report to React → User sees score dashboard + downloads fixed paper.
Section 03

Why Do We Need the LLM?

A common question: "Why can't we just use regex or rule-based code?" Here's exactly why.

❌ What Pure Code Can't Do

⚠️ Distinguish a heading from a bold paragraph — both look like bold text in the raw extracted string
⚠️ Know that "Conclusion" means the same as "Discussion" in IMRAD structure
⚠️ Match "(LeCun, 2015)" in text to "LeCun, Y. et al. (2015). Deep learning..." in references
⚠️ Detect that "[3]" and "(Wang et al., 2019)" refer to the same paper with mixed styles
⚠️ Understand that "Materials and Methods" = "Methods" section
⚠️ Extract authors from messy text like "J. Smith¹, K. Jones²·³" reliably

✅ What LLM Can Do

✓ Read context and understand document structure semantically
✓ Understand synonym sections (Conclusion ≈ Discussion)
✓ Match citations to references using fuzzy author/year matching
✓ Detect citation style inconsistencies in mixed documents
✓ Parse messy author affiliation lines correctly
✓ Produce valid structured JSON from unstructured text reliably

Which agents call the LLM and why?

Agent 1 YES — LLM — Labelling structure in text requires semantic understanding. Is "3. Methodology" a heading? Is "Abstract — " a label? Only language understanding can tell.
Agent 2 YES — LLM — Extracting structured JSON from labelled text. Counting words, detecting citation styles, identifying all unique citations. Needs language reasoning.
Agent 3 NO LLM — Just reads a JSON file from disk. This is pure Python: json.load(open("rules/apa7.json")). Fastest step in the pipeline (~0.1 sec).
Agent 4 YES — LLM (heaviest) — Must compare two complex structures, understand which elements violate rules, reformat citations, decide ordering, produce a complete sections array. Most complex reasoning task.
Agent 5 YES — LLM — Cross-referencing citations to references, detecting publication years, identifying self-citations by name matching. All need language reasoning.
Why temperature=0?

LLMs by default have randomness (temperature > 0) — same input can give slightly different outputs each run. For a formatting tool, we need 100% consistent output. Run it twice on the same paper with the same journal → get the exact same result. Setting temperature=0 makes the LLM deterministic — it always picks the highest-probability token, no randomness.

Why JSON files and not a database?
For a hackathon prototype, JSON files are fast to write, easy to read, and zero infrastructure. Adding a new journal = creating one new file. No database setup needed.
How does rule_loader.py work?
It has a JOURNAL_MAP dict with 25+ name variations mapping to filenames. "apa", "APA 7th Edition", "american psychological association" all map to "apa7.json". It loads the file, validates against a JSON schema, caches in memory.
What is FormatEngine?
engine/format_engine.py is a Python class that wraps the raw rules dict. Instead of agents doing rules["citations"]["format_three_plus"], they call engine.format_citation(["Smith","Jones","Wang"], "2020") and get back (Smith et al., 2020). It abstracts the complexity.
What happens when journal is not in the list?
rule_loader raises a RuleLoadError with the exact list of supported journals. This propagates up to FastAPI which returns HTTP 422. No silent failures.

Section 05

How Validation Works

Agent 5 runs exactly 7 independent checks. Each check scores a different aspect of the paper. Together they form the overall compliance score.

01

Citation ↔ Reference Consistency

Every in-text citation like (Smith, 2020) or [3] must have exactly one matching entry in the reference list — and vice versa. Orphan citations (cited but no reference) and uncited references (in list but never cited) are both errors. -10 per orphan citation, -5 per uncited reference.

02

IMRAD Structure Completeness

Academic papers must have Introduction, Methods, Results, and Discussion. The LLM checks if all 4 are present. Smart matching: "Materials and Methods" = Methods. "Conclusion" = Discussion. "Results and Discussion" = both. -25 per missing section.

03

Reference Age Check

Counts how many references are older than 10 years (publication year < 2015). If more than 50% are old, sets age_warning: true and adds a recommendation. Journals expect recent, relevant literature. -10 from references score.

04

Self-Citation Rate

If the paper's authors appear in more than 30% of the reference list, it flags self_citation_warning: true. Excessive self-citation is a quality concern flagged by reviewers. Adds to recommendations, -5 from references score.

05

Figure Sequential Numbering

Extracts all figure IDs (Figure 1, Figure 2, Fig. 3...), normalizes them, checks for sequential order with no gaps and no duplicates. A paper that goes Figure 1, Figure 3 (skips 2) fails this check. -15 per gap, -20 per duplicate.

06

Table Sequential Numbering

Same logic as Check 5 but for tables. Table 1, Table 2, Table 3 must be sequential with no gaps. Same deduction structure.

07

Abstract Word Count

Compares abstract.word_count from paper_structure against rules.abstract.max_words from rules JSON. APA allows max 250 words. If abstract has 267 words → flagged. -15 from abstract score. The text is NOT truncated — only flagged.

Citations 25%
Highest weight — most important for publishing
References 25%
Equally important to citations

overall_score = (citations×0.25) + (references×0.25) + (headings×0.15) + (doc_format×0.10) + (abstract×0.10) + (figures×0.075) + (tables×0.075)
submission_ready = overall_score >= 80
Section 06

Data Shapes Through the Pipeline

What type of data exists at each step of the pipeline.

crew input
→ paper_content: str (raw text, ~5,000–50,000 chars) + journal_style: str ("APA 7th Edition")
after Agent 1
→ labelled_text: str (same size + labels like [TITLE], [HEADING_H1:text] added inline)
after Agent 2
→ paper_structure: dict (JSON with title, authors, abstract, sections[], figures[], tables[], references[], metadata)
after Agent 3
→ rules: dict (JSON with document, abstract, headings, citations, references, figures, tables — 11 sections)
after Agent 4
→ transform_result: dict (violations[], changes_made[], citation_replacements[], reference_order[], docx_instructions{sections[]})
after Agent 5
→ compliance_report: dict (overall_score, breakdown{7 sections}, imrad_check, citation_consistency, recommendations[])
final output
→ formatted.docx file (python-docx writes this from docx_instructions.sections) + compliance_report (sent to React)
Two Hard Failure Rules in crew.py
Rule 1: If docx_instructions.sections is missing from Transform Agent output → ValueError raised immediately. No document written.

Rule 2: If overall_score is missing from Validate Agent output → ValueError raised immediately. No response sent.

This is the "no fallback" philosophy — if core outputs are missing, we fail loudly rather than pretending everything worked.
